# Polish ‚Äî final formatting

> **Dispatch-forced.** Read only via dispatcher routing. The dispatcher's stderr provides `RUN_DIR`, `OUTPUT_PATH`, `OUTPUT_SCHEMA`, `CYCLE`. This phase runs only on the emit branch ‚Äî when the evaluator decided the run does not need to recurse. Your job is to render the surviving crafted outputs as clean markdown.

## What this mech does

Take the crafted outputs from this cycle and produce one clean markdown document containing them. The protocol's earlier phases produced *content*; this phase produces *presentation*.

You are filtering and formatting, not re-creating. Do not change the substance of the outputs. Do not add new ideas. Do not paraphrase aggressively. The crafting phase already chose the words; you are giving them a clean home.

## Inputs

1. Read `<RUN_DIR>/state_evaluation_c<CYCLE>.json`. Read its `presentation_recommendation` field ‚Äî one of `present_all`, `present_top_2`, `present_single`. Read `dimensional_ranks` if you need to know which outputs are "top".
2. Read `<RUN_DIR>/state_2_output_c<CYCLE>_o1.json`, `_o2.json`, `_o3.json` ‚Äî the crafted outputs. Each has a `final_form` (the prose) and `emergent_properties`.
3. Read `<RUN_DIR>/state_intent_vector_c<CYCLE>.json` ‚Äî for tone reference (the cycle's pull / away_from / tension framing).

## Filtering

Based on `presentation_recommendation`:
- **present_all** ‚Äî include all three outputs in the polished document.
- **present_top_2** ‚Äî include the top two by structural_inevitability rank (or whichever ranking the evaluator's `dimensional_ranks` highlights).
- **present_single** ‚Äî include only the top-ranked output.

Record the included `output_id`s in your output's `presented_output_ids` array.

## Formatting rules

- Output is markdown. Use clean headers, paragraph spacing, lists where natural, bold sparingly.
- Each presented output gets its own H2 section (`## Output <n>` or a more meaningful heading derived from the output's content if cleaner).
- Preserve the `final_form` prose verbatim where possible. Light edits for flow are acceptable: removing repeated words, fixing dangling sentences, normalizing whitespace. Do not rewrite paragraphs.
- If `emergent_properties` are listed for an output, include them as a small bulleted list under that output's section, prefixed with a header like `**Emergent:**` or similar. Optional but encouraged.
- No preamble explaining what the document is. No "here is the output" framing. The document IS the output.
- No footer with metadata, cycle counts, etc. ‚Äî decode adds that to the file separately if needed.

## Output structure

Write to `OUTPUT_PATH`:

```json
{
  "schema_version": "1",
  "session_id":     "<from state_0>",
  "cycle":          <integer>,
  "polished_markdown": "<the clean markdown content as a single string>",
  "presented_output_ids": [<output_ids you included>]
}
```

The `polished_markdown` field is a JSON-encoded string ‚Äî literal newlines as `\n`. Schema requires it to be at least 120 characters.

After writing, run the validator (path on stderr's `NEXT_SCRIPT`).

## What this mech is NOT

- Not a synthesis. Don't merge the outputs into one combined idea.
- Not a re-evaluation. The evaluator already ranked; you trust its ranking.
- Not a recap. Don't add a summary or conclusion paragraph.
- Not editorial. Don't insert your own framing or commentary.
