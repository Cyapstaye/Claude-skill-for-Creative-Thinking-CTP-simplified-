# Shard grammar

> **Dispatch-forced.** Read this mech only when the dispatcher routes you here. The dispatcher's stderr provides:
> - `RUN_DIR` — your run's working directory
> - `NEXT_PROFILE` — the archetype name (drifter / tension_holder / remapper / compressor / wild_associator)
> - `PROFILE_CONFIG_PATH` — absolute path to the profile_config_<archetype>.json file. Read it; its `priming_preamble` is operative voice — inhabit, do not describe.
> - `OUTPUT_PATH` — where you write your `state_1_run_<archetype>.json`
> - `OUTPUT_SCHEMA` — the JSON Schema your output must satisfy
>
> Original query is in `$RUN_DIR/state_0.json` field `original_query`. Read it. Validate your written file against the schema (the validator will do this anyway; if it fails, the validator will route you back here with `RETRY=1`).

A **shard** is the atomic unit of Stage-1 output. It is a short propositional
fragment — between 20 and 160 characters — that:

- States something, rather than describing the process of arriving at it.
- Carries a structural move: a relation, an inversion, a substitution, a
  reframing. It is not a topic label and not a keyword.
- Is readable standalone. If a shard depends on an adjacent shard to be
  intelligible, it is too thin.

## What a shard is NOT

- Not a bullet-point summary of an idea.
- Not a question back to the user.
- Not a preamble like "Consider…" or "What if…".
- Not a meta-comment on the process ("the drifter is now drifting towards…").
- Not the query restated.

## Density test

Strike every adverb, every qualifier, every clause. If removing it would not
change the proposition, the shard was over-written. The COMPRESSOR profile
embodies this test; the other profiles honour it too.

## Voice

Shards are produced from inside the archetype's operative mode. The profile
does not step out to narrate. Shards read as first-order statements, not
third-order commentary.

## Count

A Stage-1 run emits between 6 and 14 shards. Fewer means the profile
under-committed; more means it diluted.

## Output structure

Write a JSON file at `$OUTPUT_PATH` matching this shape:

```json
{
  "schema_version": "1",
  "session_id":     "<read from $RUN_DIR/state_0.json>",
  "archetype":      "<from $NEXT_PROFILE>",
  "cycle":          <integer, from $CYCLE on stderr>,
  "shards":         ["<shard 1>", "<shard 2>", ..., "<shard N>"]
}
```

The validator runs `jsonschema` against `$OUTPUT_SCHEMA`. String length bounds per shard are enforced. After writing, run the validator script (path on stderr's `NEXT_SCRIPT`).
